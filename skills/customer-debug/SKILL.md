# 客户问题排查流程

> 本 skill 与 `copilot-instructions.md` 中的调试原则配合使用——后者规定"怎么做",本 skill 规定"按什么步骤做"。

---

## 第零步:demo 对照(前置动作,优先于一切假设)

**进入五步法之前先做这一步。** 这是 FAE 排查效率最高的动作,放在最前面:

1. 在 `Examples/` 或 `Firmware/` 下找出**对应外设 / 功能的官方 demo**
2. **重点比对 demo 的初始化流程**——客户问题里相当一部分根因藏在这里:
   - 时钟使能顺序(RCU 时钟 → GPIO → 外设)
   - 外设配置顺序(配参数 → 使能外设 → 使能中断/ACK 等)
   - GPIO 模式 / 复用 / 上下拉配置
   - 中断 NVIC 优先级和使能时机
   - 复位 / 软件复位的处理
3. 如果客户能提供代码 → 直接对比差异点,**差异点就是排查的第一优先方向**
4. 如果客户不给代码 → 把 demo 的初始化流程整理成"自查清单"给客户对照

**只有 demo 对照走完(或确认与 demo 完全一致)之后,才进入第一步。** 跳过这一步直接做假设,经常会绕大圈子。

### 第零步的输出格式

走完后必须按以下格式输出,便于后续步骤接力:

```
## demo 对照结果

**对照对象**:Examples/XXX/main.c (如多个外设,每个分一张表)

| 检查项 | demo 做法 | 客户做法/反馈 | 状态 |
|--------|----------|-------------|------|
| 时钟使能顺序 | 先 RCU 再外设 | 客户未回复,需自查 | 待确认 |
| 中断使能时机 | 在外设使能之后 | 客户在使能前 | **疑似偏离** |
| GPIO 复用配置 | AF7 + 上拉 | 与 demo 一致 | 一致 |

**结论**:
- 已确认偏离:N 项 → 进入第二步作为 P0 假设
- 待客户确认:M 项 → 进入第一步作为信息缺口
- 已确认一致:K 项 → 跳过,不再追究
```

无偏离时的合法输出:
> "客户实现与 demo 完全一致(已比对 N 项),第零步未发现根因线索,继续走五步法。"

找不到对应 demo 时的合法输出:
> "Examples/ 下未找到对应 XXX 的 demo,第零步跳过,直接进入第一步。
> 后续假设需要更依赖经验和 RM,置信度相应降低。"

---

## 第一步:现象确认 & 信息收集

### 1.1 复述现象
用自己的话复述客户描述,确认理解一致。复述能暴露误解。

### 1.2 列出信息缺口(三级分类)

| 级别 | 含义 | 示例 |
|------|------|------|
| **必须知道** | 没有无法推进 | MCU 型号、通信方式(轮询/中断/DMA)、是否每次复现 |
| **能缩小范围** | 可排除某些方向 | 单独测试正常否、换板/换芯片是否变化 |
| **有则更好** | 锦上添花 | 波形参数、上拉阻值、走线长度 |

给客户的问题必须是**具体是/否问题或可量化问题**,不要开放式问题。

---

## 第二步:多方向假设

每个根因按此格式列出:

```
### N. 根因描述(概率 ~XX%)

**推断依据**:根据 RM / 典型 SDK 行为 / 经验 / 波形特征 / demo 对照差异
**现象匹配**:哪些现象支持此假设
**现象矛盾**:哪些现象与假设矛盾(如有)
**验证方法**:客户能做的最简单验证
**判断类型**:demo 可验证 / 只能客户硬件验证 / 只能经验推理
```

**验证方法优先级**(从易到难):改代码 > 改配置 > 测电压 > 换元件 > 换板 > 抓波形。避免让客户做 >2 个"抓波形"级别的动作。

### 假设数量建议

**3~5 个为宜**。两端都是问题的信号:

- **少于 3 个** → 思路太窄,可能漏了大方向。**先回第零步**重新比对 demo,
  或回第一步看是否有信息缺口被忽略,再来列假设
- **多于 5 个** → 假设的区分度不够,八成把同一类问题列成多条
  (例:"DMA 配置错"和"DMA buffer 大小不对"应合并成一条 DMA 类假设,
  在该假设下分子情况列举,而不是占用两个独立坑位)

如果实在只想到 1~2 个 → **不要硬凑**,先输出现有假设并明确告诉我:
"目前只能想到 N 个方向,可能漏了 XX 维度,建议补充 YY 信息再扩展假设。"

---

## 第三步:逐步排除 & 缩小范围

### 3.1 隔离变量法

每次只改一个变量:

| 手段 | 排除什么 |
|------|---------|
| 去掉外设 | 外设间干扰 |
| 降速 | 时序裕量 |
| 换管脚 | 管脚损坏 / 冲突 |
| 换芯片 | 个体差异 |
| 换板子 | 板级问题 |
| 简化到最小复现 | 软件逻辑干扰 |
| **替换为 demo 代码运行** | **客户代码 vs demo 行为差异** |

### 3.2 三种验证思路

- **正向**:正常环境上构造异常条件 → 能复现 = 确认
- **反向**:异常环境上修复怀疑点 → 现象消失 = 确认
- **对比**:正常 vs 异常环境的差异 = 线索(客户有"好板子"和"坏板子"时最有效;**demo vs 客户代码**也是这一类的高效对比)

### 3.3 动态更新概率

每次客户给新信息:验证通过 → **明确说"已排除"**;验证失败 → 提升该方向概率;新信息引出新假设 → 加入列表。

---

## 第四步:定量分析(有数据必做)

遇到电平 / 电压 / 电流 / 时序问题时,**用公式算而不是定性猜**。

```
开漏 VOL         V_OL = V_pull × R_on / (R_on + R_pull)
反推 Ron         R_on = R_pull × V_OL / (V_pull - V_OL)
上升沿时间       t_r ≈ 2.2 × R_pull × C_bus
灌入电流         I_sink = (V_pull - V_OL) / R_pull
```

结果解读:
- 在规格内 → 正常
- **远超规格** → 硬件异常(不是容差能解释的)
- 在规格边缘 → 裕量不足

---

## 第五步:结论 & 建议

### 5.1 结论分级

| 置信度 | 表述 | 何时用 |
|--------|------|--------|
| **确认** | "根因是 XXX" | 定量数据 + 已排除其他 |
| **高度怀疑** | "大概率是 XXX" | 现象高度吻合 + 部分排除 |
| **需验证** | "怀疑 XXX,建议 YYY 验证" | 信息不足以定案 |
| **无法判断** | "基于当前信息无法确定" | 合法答案 |

### 5.2 解决方案分三类

- **临时规避**:让客户先跑起来(降速、加电阻、绕开某脚)
- **根本解决**:消除根因
- **不给方案**:信息不足时宁可不给,错误方案会让客户走弯路,损害 FAE 信誉

### 5.3 给客户的排查清单

格式模板(每条必须**动作 + 预期结果 + 含义**三要素):

```
1. 量 XXX 的电压 → 正常应是 YYY → 偏离说明 ZZZ
2. 断开 XXX 再测 → 如果变化 → 说明 ZZZ
3. 改代码 XXX   → 如果正常 → 说明 ZZZ
4. 对照 demo XXX 的初始化流程 → 如果差异在 YYY → 说明 ZZZ
```

## 五步法的非线性使用

**五步是认知顺序,不是死板的执行顺序**。真实排查会反复回流,允许甚至鼓励
以下回退/跳跃模式:

- **第二步列假设时,发现某些现象解释不了** → **回第一步**补充信息缺口,
  不要硬把现象塞进现有假设
- **第三步验证排除了所有假设** → **回第二步**重新发散假设,可能整个方向错了
- **第三步反复无进展 + 客户已疲惫** → **回第零步**重做一次 demo 对照,
  很可能漏了某个维度
- **第四步定量发现"远超规格"** → 直接跳到第五步给"硬件异常"结论,
  不必硬走完第三步
- **客户在中途给出关键新信息**(例:"换了电源就好了") → 重新评估,
  之前所有假设的概率都要重排,不要为"流程完整"硬走原路径

**核心原则:效率优先,不为流程完整硬走五步**。
但每次回流都明确告诉我:"基于 XX 信息,我从第 N 步回到第 M 步,
原因是 YY"——让我看到推理路径,而不是默默重启。

## 与专项 skill 的关系

本 skill 是通用方法论。遇到特定外设问题时 Copilot 应同时加载对应专项 skill(如 `i2c-debug` / `usb-debug` / `ethernet-debug` / `toolchain-debug`)。专项 skill **只补充外设特有经验**,五步法流程以本 skill 为准。
