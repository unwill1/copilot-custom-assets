# 理解客户代码模式

本次任务:帮我快速吃透一份客户工程代码。客户基于我们原厂固件库 API 写的应用层,
通常比较复杂。

---

## 模式判断 & 调度

根据我下面的描述自动选择路径:

### 默认路径:模式 A — 架构理解

我只想搞清楚这工程怎么跑起来、怎么实现功能。**继续往下看本 prompt 的模式 A 部分**。

### 触发路径:走 `customer-debug-flow` skill

我的描述里出现以下任一关键词时,**不走本 prompt 的模式 A,直接调用
`customer-debug-flow` skill 的五步法**(本 prompt frontmatter 已 reference):

- "有问题 / 有 bug / 异常 / 不工作 / 跑飞 / 挂死 / 偶发"
- "客户反馈 / 客户报告 / 现场 / 复现 / 失败"
- "帮我看看哪里不对 / 找 bug"
- 描述具体异常现象(不通信、丢数据、复位、性能差等)

**唯一前置约束**:如果客户工程代码量大、你还没建立对工程的基本理解,
**先回到模式 A 完成理解再进 debug**——不要在没看懂的代码上硬 review。
模式 A 完成后我会显式说"现在进 debug"或重新触发 explain,你才进 skill。

### 兜底:无法判断时先问

关键词没命中、但描述里又有疑似异常的措辞时,**先问我一句**:
"你是想先理解工程,还是已经发现问题要排查?"
不要自行猜测进哪条路——猜错的代价(白做一遍架构图 / 白进 debug 流程)
比多问一句大得多。

### 进入路径前先声明

无论走哪条路,**先用一句话声明你判断的路径和依据**,再开始实际工作:

- "→ 走模式 A(架构理解),你的描述未提到具体异常"
- "→ 走 customer-debug-flow skill,触发关键词:'客户反馈通信失败'"
- "→ 暂停,需要先问你一个问题:..."

让我能在判断错的第一时间纠正,而不是等你输出完一整套东西才发现走错。

## 模式 A:架构理解(默认)

### 开始前:先报告工程规模

进入产出之前,先快速扫一遍工程,报告规模:

- 大致代码量(多少个 .c/.h 文件、估算多少行业务代码,排除 Firmware/)
- 主要文件清单(top 5~10 个看起来最关键的)
- 软件架构初判(裸机 / RTOS / 中间件)

如果工程规模 **> 50 个源文件 或 > 10K 行业务代码**,**先停下来问我聚焦哪部分**:

- (A) 全工程概览(粗粒度,可能漏细节)
- (B) 特定模块(我指定哪个 .c/目录)
- (C) 特定功能链路(我描述一个功能,你追这条链路)

不要无差别全扫一遍——大工程全扫会输出过度泛泛的概览,等于没说。
小工程(< 50 文件)直接进下一步,不用问。

### 产出内容(按以下顺序输出)

#### 1. 工程概览(3-5 句话)

- 这是一个什么功能的工程
- 用了哪些核心外设 / 中间件
- 软件架构风格(裸机轮询 / 裸机中断 / RTOS / 状态机 / 事件驱动)

#### 2. 模块结构图

由你判断用什么图最合适(flowchart / sequenceDiagram / stateDiagram / classDiagram /
component diagram / ASCII 框图)。判断依据:

- 模块间是**调用关系** → flowchart 或 component diagram
- 模块间是**消息/事件传递**(尤其 RTOS) → sequenceDiagram
- 主体是**状态机** → stateDiagram
- 多个并发任务 → 用 sequenceDiagram 或分泳道
- 简单工程 → ASCII 框图够用,不用上 Mermaid

**优先 Mermaid**(我装了渲染插件)。如果 Mermaid 表达不出来,再退回 ASCII。

每张图配一段简短文字说明(图里看不出来的关键信息)。

需要多张图分层时:**先顶层架构,再关键流程**。不要一张图试图说清所有事。

#### 3. 关键路径追踪:从 main() 跑起来

按程序执行顺序追踪:

```
main()
 ├─ 初始化阶段
 │   ├─ 时钟配置 → ...
 │   ├─ 外设初始化 → 用了哪些原厂 API
 │   └─ 应用模块初始化 → 谁初始化了什么
 │
 ├─ (如有 RTOS) 任务创建
 │   ├─ TaskA: 干什么的
 │   └─ TaskB: 干什么的
 │
 └─ 主循环 / 调度器启动
     └─ 主循环里在做什么
```

**重点说清楚"程序是被什么驱动的"**:定时器中断?外部事件?RTOS 调度?
轮询?这是理解工程行为的关键。

#### 4. 中断 / 事件入口清单

列出所有中断 ISR 和事件回调:

| 入口 | 文件:行 | 触发源 | 干什么 | 和主循环/任务的交互 |
|------|--------|--------|--------|--------------------|
| USART0_IRQHandler | xxx.c:123 | 串口接收 | 收数据放队列 | 通过 ring buffer 给 TaskA |

文件路径用**工作区相对路径**,VS Code 可点击跳转。

#### 5. 与 Examples 流程对比

在 `Examples/` 下找出**对应外设 / 功能最接近的官方例程**作为基准,对比客户代码
的实现差异。这是 FAE 视角最重要的一节 —— 大部分客户问题就藏在差异里。

对比维度(每个外设都过一遍):

| 维度 | 例程做法 | 客户做法 | 差异性质 |
|------|---------|---------|---------|
| 时钟使能顺序 | ... | ... | 相同 / 改写 / 缺失 / **疑似错** |
| GPIO 配置 | ... | ... | ... |
| 外设初始化参数 | ... | ... | ... |
| 中断使能时机 | ... | ... | ... |
| 数据收发方式(轮询/中断/DMA) | ... | ... | ... |
| 错误处理 | ... | ... | ... |

**差异性质**用四档:

- **相同**:客户和例程一致,跳过
- **改写**:做法不同但合理(比如客户用了不同管脚 / 改了优先级)
- **缺失**:例程有的步骤客户没做(高风险,重点关注)
- **疑似错**:看起来违反例程的关键顺序或推荐做法(最高优先级)

**只列差异,不展开分析**。"缺失"和"疑似错"的项目自动进入下一节"可疑点"。

如果客户工程涉及多个外设,**每个外设单独对比**,不要混在一起。

##### 找不到对应例程时:直接说,不要硬凑

如果在 `Examples/` 下**找不到合适的对应例程**(功能差异太大 / 客户用了
组合外设 / 客户用了我们没出例程的特性),**直接说明并跳过对比表**。

不要硬找一个不相关的例程对比——那会产生**误导性的"差异"**,
让我把"客户做了创新"误判成"客户偏离了规范"。

合法的回复格式:
> "本工程使用的 `XXX + YYY` 组合在 Examples 下没有直接对应例程。
> 最接近的是 `Examples/AAA`,但只覆盖 XXX 部分。
> 因此**只对 XXX 部分做差异分析**,YYY 部分跳过对比直接进可疑点环节。"

部分对应也是合法的——对得上的部分按表对比,对不上的部分跳过并说明。

#### 6. 可疑点 / 风险点(只列不展开)

可疑点有两个来源,**用标签区分**,方便后续判断怎么和客户沟通:

- **[对比]** ——来自第 5 步对比表里所有标"缺失"和"疑似错"的项。**自动进入此清单,不要漏**。这类问题对客户的话术是"按我们例程的推荐做法应该是 XX"
- **[审阅]** ——读代码过程中发现的、与例程对比无关的疑点(ISR 阻塞、共享变量未加 volatile、DMA 对齐、栈溢出风险等)。这类问题对客户的话术是"这是嵌入式通用最佳实践"

```
## 可疑点(待 review,本次不展开)

- [ ] [对比] xxx.c:80 — 例程在使能 USART 前 enable RX 中断,客户顺序相反
- [ ] [对比] yyy.c:30 — 例程使能了 DMA TC 中断,客户工程里缺失
- [ ] [审阅] xxx.c:45 — ISR 里调用了 printf,可能阻塞
- [ ] [审阅] yyy.c:120 — 共享变量 g_flag 没加 volatile
- [ ] [审阅] zzz.c:200 — DMA buffer 没看到对齐声明(M7 平台需要 cache 对齐)
```

每条:**[来源] + 位置 + 一句话描述疑点**。不要在这一步给修复建议,不要展开分析。

如果我后面想 review,可以说"针对第 N 条展开" 或 "进入 review 模式"。

## 通用原则(两个路径都适用)

- **代码引用必须是可点击的 Markdown 链接**:`[xxx.c:45](path/xxx.c)`,工作区相对路径
- **不确定的不要编造**:看不懂某段代码、不确定某个 API 行为、不确定某段逻辑的意图
  → 先问我,不要脑补
- **遇到客户用的原厂 API 不熟悉的**:去 `Firmware/` 头文件确认,而不是凭印象推断
- **保持简洁直接**,不要"这是一个很好的问题"之类铺垫
