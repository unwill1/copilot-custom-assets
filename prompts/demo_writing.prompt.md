# Demo 编写模式

本次任务:基于已打开的**基础工程模板或参考例程**做新 demo,或在已有工程上
增删改功能。**不是排查问题,不是从零创建工程。**

我可能传入了具体需求(`${input:requirement}`),也可能没传。无论哪种,
都按下面的流程走。

---

## 强制工作流(四步,不可跳)

### 第一步:阅读参考例程

动手写代码之前先做以下事:

1. 浏览 `#codebase` 中 `Examples/` 目录,找出和需求**最接近的例程**作为参考蓝本
2. **重点理解例程的实现流程**:
   - main 函数的整体执行路径
   - 初始化顺序(时钟 → GPIO → 外设 → 中断 → 启动)
   - 数据流(谁产生数据、谁搬运、谁消费)
   - 错误处理模式
3. 浏览 `Firmware/` 下相关外设的头文件,确认**实际可用的 API**
   - 不要凭印象写"看起来像"的函数名
   - 不确定 API 是否存在 → 必须查头文件,查不到就问我

**例程的实现风格就是这次 demo 的实现风格**,不要自创规范。

#### 第一步完成后:必须先做反向确认

读完例程**不要直接进第二步**。先用 2-3 句话**复述参考例程的核心结构**,
让我确认你确实读懂了:

```
参考例程:Examples/XXX/main.c

- 文件结构:main.c 负责 X,xxx_it.c 负责 Y,yyy.c 是 Z 模块
- main 流程:时钟初始化 → GPIO → USART(中断模式) → 主循环空转等中断
- 关键外设配置:USART1 @ 115200,RX 中断优先级 2,无 DMA
- 数据流:RX 中断收字节 → ring buffer → 主循环消费
```

格式不限,但必须包含**文件分工、main 流程、关键外设配置、数据流**这四点。
我确认后再进第二步。

#### 如果发现例程不合适:直接说,不要硬凑

读完例程后如果判断**这个例程不适合作为参考蓝本**(差太远 / 缺关键外设 /
风格不匹配 / 思路完全不同),**直接告诉我**,我重新指定参考例程。

不要在错的蓝本上硬改——蓝本错了,后面三步全是浪费。
合法的回复是:"这个例程不太合适,原因是 XX,建议参考 YY 或者你重新指定。"

---

### 第二步:一次性反问清楚需求(最关键)

我和客户的需求描述**几乎一定不够明确**。在动手前把所有模糊点列出来
**一次性问完**,等我全部回答后再开工。不要边写边问。

#### 必问清单(每次都要过一遍)

按以下维度逐项检查需求,凡是没明说的都要问。**不是每次都全问完**——
和本次需求无关的维度跳过,但每个维度都要过一眼,确认是不是真的无关。

**功能层**
- 输入(来源/接口) → 输出(目的地/接口) → 触发方式(轮询/事件/周期/手动)
- 异常情况怎么处理(超时、错误、越界、资源不足)

**外设配置层**
- 速率/频率/波特率/分辨率等**具体数值**
- 工作模式(主从、全/半双工、单次/连续、对齐方式)
- 引脚分配(用例程默认还是改管脚) + 时钟源(HSI/HSE/PLL/LSE)

**实现方式层**
- **轮询 / 中断 / DMA —— 永远要问,默认无法假设**
- 中断优先级(用中断时) / DMA 模式与双 buffer(用 DMA 时)
- 缓冲区大小、是否要 ring buffer / 队列、阻塞 vs 非阻塞接口

**数据层**
- 数据格式(字节序、对齐、packed) / buffer 大小、最大帧长
- 是否需要校验(CRC / checksum / parity)

**接口层**
- 写成可复用库函数还是 main 里直接调用
- 函数签名(返回值类型、错误码风格、回调注册方式) / 是否需要状态查询接口

**资源约束**
- RAM / Flash 限制 / 和现有外设是否冲突(管脚、DMA 通道、定时器)
- 功耗要求(影响是否能用 sleep / 关时钟)

**测试 / 验证**
- 怎么验证 demo 工作正常(LED / 串口打印 / 示波器 / ...) / 是否要内置自测

#### 反问格式

按以下格式列出问题,**带编号 + 候选项**,方便我快速回复:

```
开工前需要确认以下 N 点,请逐项回复:

1. **数据触发方式**:
   - (A) 定时器周期触发
   - (B) 外部 GPIO 中断触发
   - (C) 主循环主动调用
   倾向:(B),因为你提到"按键按下时采样",但需要你确认

2. **DMA 是否使用**:
   - (A) 用 DMA(适合连续大量数据)
   - (B) 不用 DMA,中断里直接搬
   推荐:(A),数据量 1KB 以上 DMA 更稳

3. ...
```

每个问题:
- 给出 2-4 个候选项,而不是开放式问
- 如果有合理默认 / 推荐,**说明并给出理由**(让我快速 OK 或否决)
- 如果某个点你已经能从我的描述推断,**说出来让我确认**,而不是直接采用

### 第三步:等我回答后,给出改动清单(先不出代码)

我回完问题后,**先不要直接写代码**。先给我一份**改动清单**:

```
## 改动清单(待确认)

基于例程:Examples/XXX/main.c

### 新增
- [ ] Examples/XXX/myfunc.c — 新增模块,实现 YYY 功能
- [ ] main.c 第 X 行 — 调用 myfunc_init()
- [ ] main.c 主循环内 — 加 YYY 处理

### 修改
- [ ] main.c 第 X 行 — 时钟配置改 ZZZ(原因:...)
- [ ] gd32xxx_it.c — 加 USARTx_IRQHandler 实现

### 删除
- [ ] main.c 第 X~Y 行 — 原例程的 demo 代码不再需要

### 涉及的固件库 API
- gpio_mode_set / gpio_output_options_set
- usart_dma_enable / dma_channel_enable
- 已确认全部存在于 Firmware/ 中

### 明确不动的部分(防止顺手改)
- 例程的时钟初始化函数 — 只读不改
- 例程现有的中断处理框架 — 只加新 handler,不动现有的
- 例程的 GPIO 初始化 — 新外设引脚另开函数,不合并到原函数

### 风险点
- DMA 通道 X 是否和现有 ADC 冲突 —— 需要你确认
```

文件路径用**工作区相对路径**,VS Code 里可点击跳转。

> 关于"明确不动的部分":除了我同意修改的内容,例程里其他**能跑的代码一律不动**。
> 看到风格不一致、有更优写法、注释不规范——**提一嘴可以,动手不行**。
> 这是 demo,不是重构。

确认改动清单后,再进入第四步出代码。这一步的目的:让我看到你打算怎么动工程,
有问题趁早拦截,避免写完一大坨才发现方向错。

- 回复末尾列改动清单(`文件 : 改了什么(行号)`)
